## II. THE RAMSAY (Design Discrepancies)

> "This is raw! You're telling me this is the finished product?"

These are not typos. These are places where the spec describes one design and the
proven codebase implements a different (often better) design. An implementer who
follows the spec literally would produce working code that contradicts the
existing pattern -- creating two incompatible approaches in the same system.

### D-1: AccountId Extraction Pattern

| Aspect | What the Docs Say | What Openfang Actually Does |
|--------|-------------------|-----------------------------|
| Mechanism | `extract_account` async middleware function injecting `AccountContext` into request extensions | `impl FromRequestParts for AccountId` -- an Axum extractor, not middleware |
| Type | `AccountContext { account_id: String, source: AccountSource }` | `AccountId(Option<String>)` -- no source tracking |
| Integration | Added to middleware chain between `bearer_auth` and `rate_limit` | Added as a handler parameter, extracted automatically by Axum's type system |

> **FIXED (2026-04-07):** MASTER-PLAN now uses `FromRequestParts` pattern throughout.
> `AccountContext` references replaced with `AccountId(Option<String>)`.

**Impact:** The `FromRequestParts` approach is cleaner, more testable, and
already proven across 50+ handlers in openfang-ai. The middleware approach
described in the docs would work but introduces unnecessary coupling. The
spec should be updated to match the proven pattern, or the spec should
explicitly flag this as a deliberate divergence with rationale.

### D-2: Timestamp Replay Protection -- Spec Fiction

ADR-MT-002 and MASTER-PLAN describe a timestamp-based replay protection scheme:

```
X-Account-Id: acct_abc123
X-Account-Signature: hmac(account_id + timestamp, secret)
X-Account-Timestamp: 1712419200
```

With a +/- 5 minute tolerance window.

**Reality in qwntik:** The HMAC is computed over `accountId` alone. There is no
`X-Account-Timestamp` header. There is no replay window. The signature header is
`X-Account-Sig` (not `X-Account-Signature`).

This is not a bug in qwntik -- it is a simpler, working design. The spec either
needs to:

1. Drop timestamp protection and document why (simplicity, statelessness), or
2. Flag it explicitly as "planned enhancement, not yet implemented in qwntik"

Currently the spec reads as if this is a proven pattern being ported. It is not.

### D-3: Header Name Mismatch

| Document | Header Name Used | Actual Header in Qwntik |
|----------|-----------------|-------------------------|
| MASTER-PLAN, ADR-MT-002 | `X-Account-Signature` | `X-Account-Sig` |
| MIGRATION-GUIDE | `X-Account-Timestamp` | Does not exist |

If a developer implements the MIGRATION-GUIDE examples verbatim, the requests
will fail against qwntik because the header names are wrong.

### D-4: Vector Operations -- Edge Functions vs RPCs

Some docs reference "edge functions" for vector-search/upsert operations. Qwntik
actually uses PostgreSQL RPCs via `supabase.rpc()`, not Supabase Edge Functions.
These are fundamentally different execution models:

- **Edge Functions**: Deno runtime, deployed separately, invoked via HTTP
- **RPCs**: SQL functions called directly through PostgREST, no separate deployment

An implementer who builds edge function wrappers would be doing unnecessary work.

### D-5: MemoryFilter Account Scoping

ADR-MT-004 proposes adding `account_id` to the `MemoryFilter` struct. In
openfang-ai, `MemoryFilter` does NOT have an `account_id` field. Account
filtering happens at the SQL query level (WHERE clause injection), not at the
filter struct level. This means:

- The filter struct stays generic and reusable
- Account scoping is enforced at the data access layer, not the domain layer
- You cannot accidentally bypass account filtering by constructing a raw filter

The openfang approach may actually be superior (defense in depth at the query
layer), but the spec should acknowledge this divergence and justify whichever
approach LibreFang adopts.

### Assessment

The document set is **architecturally sound but factually unreliable in spots**.
The design decisions are good. The phase decomposition is realistic. The
backward-compatibility story is well thought out. The openfang pattern mining
is genuine where verified.

The problem is that the docs were written with a mix of verified facts and
assumptions-stated-as-facts. When a document says "233+ SQL functions" and the
real number is 161, that is not a rounding error -- it is a 45% overcount that
would fail any automated acceptance test. When a document describes
`AccountContext` as a "proven pattern from openfang" and openfang has never had
an `AccountContext` struct, that erodes trust in every other "proven pattern"
claim.

### Recommendations

1. **Apply the Priority 1 corrections immediately.** These are find-and-replace
   operations. Ten minutes of work. No architectural judgment required.

2. **Resolve the design discrepancies (D-1 through D-5) before Phase 1
   implementation begins.** Each one needs a decision: match openfang, or
   diverge with documented rationale. Do not leave both patterns in the docs.

3. **Add a "Verified" stamp to each document section.** After corrections, mark
   each section with the date and method of verification (e.g.,
   "Verified 2026-04-07 via grep against openfang-ai@main"). This creates an
   audit trail and makes future BHR runs incremental rather than full-sweep.

4. **Run a BHR v3 fix pass** targeting the 7 open critical/high items from
   this report. Then re-audit. The goal is zero critical findings before any
   code is written against these specs.
