### C-002 — go-env-greet fixture rebuild with correct WASI P1 adapter

**Branch:** same worktree
**Files:**
- `xtask/src/plugins.rs` (update `build_go_env_greet` to resolve adapter
  from `wasi-preview1-component-adapter-provider@45.0.0` in cargo registry)
- `Cargo.toml` (add `wasi-preview1-component-adapter-provider = "45.0.0"` as
  a dev-dep so `cargo fetch` downloads it)
- `xtask/Cargo.toml` (forward the dep)
- `examples/plugins/go-env-greet/pre-built/plugin.wasm` (rebuilt artefact)
- `examples/plugins/go-env-greet/go.sum` (regenerated by go mod tidy)

**Depends on:** none (independent of C-003)
**Effort:** M
**Recommended agent:** `claude`

#### Why this works

`wasi-preview1-component-adapter-provider 45.0.0` bundles the exact adapters
from the wasmtime v45 release. The crate exposes its artefact paths via a
`CARGO_MANIFEST_DIR`-relative constant or directly at the well-known path
`~/.cargo/registry/src/<hash>/wasi-preview1-component-adapter-provider-45.0.0/artefacts/wasi_snapshot_preview1.reactor.wasm`.
The xtask builder reads this path at build time — no committed binary, no
network, version locked by the workspace.

#### Tasks

- [ ] Add `wasi-preview1-component-adapter-provider = { version = "45.0.0", optional = true }`
  (or dev-dep) to root `Cargo.toml` under `[workspace.dependencies]`.
  Forward to `xtask/Cargo.toml`.
- [ ] In `xtask/src/plugins.rs::build_go_env_greet`, replace the
  hard-coded `dir.join("wasi-adapter/wasi_snapshot_preview1.reactor.wasm")`
  adapter path with a helper that locates the adapter in the cargo registry:
  ```rust
  fn wasi_reactor_adapter_from_registry() -> XtaskResult<PathBuf> {
      // The crate exposes the adapter path via its CARGO_MANIFEST_DIR
      // and the artefacts/ subdir. We locate it via CARGO_HOME.
      let cargo_home = std::env::var("CARGO_HOME")
          .map(PathBuf::from)
          .unwrap_or_else(|_| home::home_dir().unwrap().join(".cargo"));
      let pattern = cargo_home
          .join("registry/src/*/wasi-preview1-component-adapter-provider-45.0.0/artefacts/wasi_snapshot_preview1.reactor.wasm");
      glob::glob(pattern.to_str().unwrap())
          .map_err(|e| format!("glob: {e}"))?
          .find_map(|r| r.ok())
          .ok_or_else(|| "wasi-preview1-component-adapter-provider@45.0.0 not in cargo registry — run `cargo fetch` first".into())
  }
  ```
  Uses the `glob` crate (already in workspace via other deps) or falls back
  to scanning registry dirs manually.
- [ ] Run `cargo fetch` to pull `wasi-preview1-component-adapter-provider 45.0.0`
  into the local registry.
- [ ] Rebuild: `TINYGOROOT=/tmp/tinygo cargo xtask plugins-rebuild go-env-greet`
- [ ] Verify: `wasm-tools dump examples/plugins/go-env-greet/pre-built/plugin.wasm | grep _initialize` — confirms reactor mode.
- [ ] Verify: `wasm-tools component wit examples/plugins/go-env-greet/pre-built/plugin.wasm` — shows correct imports.

**Done when:** xtask builder uses registry adapter, `cargo xtask
plugins-rebuild go-env-greet` succeeds, rebuilt component exports
`_initialize` (not `_start`), and `wasm-tools validate` passes.

### C-003 — js-kv-counter fixture rebuild with WASI 0.2.6-compatible componentize-js

**Branch:** same worktree
**Files:**
- `xtask/src/plugins.rs` (update `build_js_kv_counter` to use `npx
  @bytecodealliance/componentize-js@0.19.4` instead of global `jco
  componentize`)
- `examples/plugins/js-kv-counter/pre-built/plugin.wasm` (rebuilt artefact)

**Depends on:** none (independent of C-002)
**Effort:** M (with verify-first)
**Recommended agent:** `claude`

#### Why 0.19.4

`componentize-js 0.19.x` predates the WASI 0.2.10 adoption. The
`@0.2.10` WASI imports in the current js-kv-counter were introduced when
jco 1.20 / componentize-js 0.21 bumped their embedded StarlingMonkey. Using
0.19.4 produces a component with `wasi:io/poll@0.2.6` (or earlier) which
wasmtime-wasi 45 can serve without ABI mismatch.

**Fallback:** if 0.19.4 still produces 0.2.10 imports, try 0.18.5.

#### Tasks

- [ ] **Verify-first smoke**: run `npx --yes @bytecodealliance/componentize-js@0.19.4 --help` to confirm the CLI exists and downloads cleanly via npx.
- [ ] Update `build_js_kv_counter` in `xtask/src/plugins.rs` to invoke
  `componentize-js` via npx rather than `jco componentize`:
  ```
  npx --yes @bytecodealliance/componentize-js@0.19.4
      app.js
      --wit <wit>
      --world-name plugin
      --out-dir <tmpdir>
  ```
  Note: `componentize-js` CLI writes to a directory, not a single file;
  move the resulting `app.wasm` to `pre-built/plugin.wasm`.
- [ ] Run rebuild: `cargo xtask plugins-rebuild js-kv-counter`
- [ ] Verify WASI version: `wasm-tools component wit examples/plugins/js-kv-counter/pre-built/plugin.wasm | grep wasi:io/poll`
  — must show `@0.2.6` (or earlier), NOT `@0.2.10`.
- [ ] If still 0.2.10: fall back to `componentize-js@0.18.5` and document in
  change record.

**Done when:** `wasm-tools component wit js-kv-counter/pre-built/plugin.wasm`
shows `wasi:io/poll@0.2.6` (or earlier) and xtask size-budget check passes.
